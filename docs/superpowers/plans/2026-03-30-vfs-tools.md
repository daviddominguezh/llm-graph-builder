# VFS Tools Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement the 12 VFS tool functions that wrap VFSContext methods, plus tool registration, response helpers, symbol patterns, and the system prompt preamble.

**Architecture:** Each tool is a thin adapter in `packages/api/src/vfs/tools/` — validate input via Zod schema, call VFSContext, map result to snake_case response. A `toolResponse.ts` module provides shared `toToolSuccess`/`toToolError` helpers. Tools are registered via `generateVFSTools(context, vfs)` and merged into the tool map when VFS is enabled.

**Tech Stack:** TypeScript (ESM, strict), Zod (input schemas), Vercel AI SDK `tool()` function, picomatch (already installed)

**Spec:** `docs/superpowers/specs/2026-03-30-vfs-tools-design.md`

**Depends on:** Spec 1 (VFS Core) — all VFSContext methods, types, VFSError

**ESLint constraints:** max 40 lines/function, max 300 lines/file, max depth 2.

### Task 1: Tool response helpers and tool enum

The response helpers bridge VFSContext results (camelCase) to the AI SDK's `ToolResponsePrompt` format. The success helper wraps a structured object at `result.result`; the error helper wraps a `VFSError` into `{ success: false, error, error_code, details }` with `isError: true`.

**toolResponse.ts** — two functions:

```typescript
import type { ToolResponsePrompt } from '@src/types/tools.js';
import { VFSError } from '../types.js';

export function toToolSuccess<T>(toolCallId: string, toolName: string, data: T): ToolResponsePrompt {
  return {
    type: 'tool-result',
    toolCallId,
    toolName,
    result: { result: { success: true, ...data } },
  };
}

export function toToolError(toolCallId: string, toolName: string, error: VFSError): ToolResponsePrompt {
  return {
    type: 'tool-result',
    toolCallId,
    toolName,
    isError: true,
    result: {
      result: {
        success: false,
        error: error.message,
        error_code: error.code,
        ...(error.details !== undefined ? { details: error.details } : {}),
      },
    },
  };
}
```

**Tests** should verify:
1. `toToolSuccess` wraps data with `success: true` at `result.result`
2. `toToolError` wraps VFSError with `success: false`, `isError: true`, correct field mapping
3. `toToolError` omits `details` when VFSError has no details

### Task 2: Zod schemas for all 12 tools

**Files:**
- Create: `packages/api/src/vfs/tools/schemas.ts`

All 12 tool input schemas in one file (stays DRY — shared imports, consistent patterns). Use snake_case field names since these are the wire format the LLM sends.

**Schemas:**

```typescript
import z from 'zod';

export const ReadFileSchema = z.object({
  path: z.string().describe('Relative path from repo root'),
  start_line: z.number().int().min(1).optional().describe('1-based start line'),
  end_line: z.number().int().min(1).optional().describe('1-based end line (inclusive)'),
});

export const ListDirectorySchema = z.object({
  path: z.string().default('').describe('Directory path, defaults to repo root'),
  recursive: z.boolean().default(false),
  max_depth: z.number().int().min(1).default(2),
});

export const FindFilesSchema = z.object({
  pattern: z.string().describe('Glob pattern, e.g. "**/*.ts"'),
  path: z.string().optional().describe('Directory scope'),
  exclude: z.array(z.string()).optional().describe('Glob patterns to exclude'),
  max_results: z.number().int().min(1).default(100),
});

export const SearchTextSchema = z.object({
  pattern: z.string().describe('Search string or regex'),
  is_regex: z.boolean().default(false),
  path: z.string().optional().describe('Directory scope'),
  include_glob: z.string().optional().describe('Only search matching files'),
  ignore_case: z.boolean().default(false),
  max_results: z.number().int().min(1).default(50),
});

export const GetFileMetadataSchema = z.object({
  path: z.string(),
});

export const GetFileTreeSchema = z.object({
  path: z.string().default(''),
  max_depth: z.number().int().min(1).default(3),
});

export const CountLinesSchema = z.object({
  path: z.string(),
  pattern: z.string().optional(),
  is_regex: z.boolean().default(false),
});

export const SearchSymbolSchema = z.object({
  name: z.string().describe('Symbol name or prefix'),
  kind: z.enum(['function', 'class', 'interface', 'variable', 'type', 'any']).default('any'),
  path: z.string().optional(),
});

export const CreateFileSchema = z.object({
  path: z.string(),
  content: z.string(),
});

export const EditFileSchema = z.object({
  path: z.string(),
  edits: z.array(z.object({
    old_text: z.string(),
    new_text: z.string(),
  })).optional(),
  full_content: z.string().optional(),
}).refine(
  (data) => (data.edits !== undefined) !== (data.full_content !== undefined),
  { message: 'Provide either edits or full_content, not both and not neither' }
);

export const DeleteFileSchema = z.object({
  path: z.string(),
});

export const RenameFileSchema = z.object({
  old_path: z.string(),
  new_path: z.string(),
});
```

Each tool file follows the same pattern:

```typescript
import { tool } from 'ai';
import type { VFSContext } from '../vfsContext.js';
import { VFSError } from '../types.js';
import { ReadFileSchema } from './schemas.js';
import { VFSTool } from './toolEnum.js';
import { toToolError, toToolSuccess } from './toolResponse.js';

export function createReadFileTool(vfs: VFSContext) {
  return tool({
    description: 'Read file content. Use start_line/end_line for large files.',
    inputSchema: ReadFileSchema,
    execute: async (data, { toolCallId }) => {
      try {
        const result = await vfs.readFile(data.path, data.start_line, data.end_line);
        return toToolSuccess(toolCallId, VFSTool.read_file, {
          path: result.path,
          content: result.content,
          start_line: result.startLine,
          end_line: result.endLine,
          total_lines: result.totalLines,
          token_estimate: result.tokenEstimate,
        });
      } catch (error) {
        if (error instanceof VFSError) return toToolError(toolCallId, VFSTool.read_file, error);
        throw error;
      }
    },
  });
}
```

**Key points per tool:**
- Each tool maps VFSContext camelCase results to snake_case response
- Each catches VFSError and returns toToolError; rethrows unknown errors
- Short descriptions (one sentence) — cross-tool guidance goes in preamble
- `list_directory`: returns `{ path, entries: [{name, type}] }`
- `find_files`: returns `{ pattern, matches, total_matches, truncated }`
- `search_text`: passes params object to `vfs.searchText(params)`, maps match fields
- `get_file_metadata`: returns `{ path, size_bytes, line_count, language, is_binary }`
- `get_file_tree`: passes `maxDepth` from `data.max_depth`, returns `{ path, tree, truncated }`
- `count_lines`: returns `{ path, total_lines, matching_lines?, pattern? }`

3. Maps camelCase to snake_case correctly

**symbolPatterns.ts** — regex patterns per language for finding definitions:

```typescript
interface SymbolPattern {
  kind: string;
  regex: RegExp;
}

// JS/TS patterns
const JS_TS_PATTERNS: SymbolPattern[] = [
  { kind: 'function', regex: /^\s*(?:export\s+)?(?:async\s+)?function\s+(\w+)/ },
  { kind: 'function', regex: /^\s*(?:export\s+)?const\s+(\w+)\s*=\s*(?:async\s+)?\(/ },
  { kind: 'function', regex: /^\s*(?:export\s+)?const\s+(\w+)\s*=\s*(?:async\s+)?(?:\([^)]*\)|[^=])\s*=>/ },
  { kind: 'class', regex: /^\s*(?:export\s+)?(?:abstract\s+)?class\s+(\w+)/ },
  { kind: 'interface', regex: /^\s*(?:export\s+)?interface\s+(\w+)/ },
  { kind: 'type', regex: /^\s*(?:export\s+)?type\s+(\w+)\s*=/ },
  { kind: 'variable', regex: /^\s*(?:export\s+)?(?:const|let|var)\s+(\w+)\s*[=:]/ },
];

// Python patterns
const PYTHON_PATTERNS: SymbolPattern[] = [
  { kind: 'function', regex: /^\s*(?:async\s+)?def\s+(\w+)/ },
  { kind: 'class', regex: /^\s*class\s+(\w+)/ },
  { kind: 'variable', regex: /^(\w+)\s*=/ },
];

// Go patterns
const GO_PATTERNS: SymbolPattern[] = [
  { kind: 'function', regex: /^func\s+(?:\([^)]+\)\s+)?(\w+)/ },
  { kind: 'type', regex: /^type\s+(\w+)\s+(?:struct|interface)/ },
  { kind: 'variable', regex: /^(?:var|const)\s+(\w+)/ },
];
```

Export a function `getLanguagePatterns(language: string): SymbolPattern[]` that returns patterns for the language, or empty array for unsupported languages.

Export a function `findSymbols(content: string, language: string, name: string, kind: string): SymbolMatch[]` that scans lines, applies patterns, and returns matches where the symbol name starts with the search name (prefix matching).

**searchSymbol.ts** — the tool wraps this. It needs to:
1. Ensure tree is loaded (call `vfs.getFileTree()` to trigger tree loading)
2. Get candidate files by language from tree
3. For each candidate, read the file and run `findSymbols`
4. Return matches

Note: `VFSContext.searchSymbol` is currently a stub. The tool should call VFSContext methods directly (readFile + tree) rather than relying on the stub. Alternatively, update VFSContext.searchSymbol to accept `kind` and `path` params and implement it properly using the symbolPatterns helpers.

**Recommended approach:** Implement the symbol search logic in the tool file itself (calling `vfs.readFile` and `vfs.findFiles` for candidate discovery), keeping VFSContext's `searchSymbol` as-is for now. The tool knows about languages and patterns; VFSContext doesn't need to.
